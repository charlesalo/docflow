# Raw AI Usage Log

Chronological, unpolished log of the Claude Code session that built this project. See `AI_WORKFLOW.md` for the summarized version required by the brief.

1. Read `Assessment_Brief.md` and `CLAUDE.md` for scope/time constraint, inspected the pre-existing empty scaffold (`app/`, `docs/`, `notes/`, placeholder docs).
2. Scaffolded Next.js (App Router, TS, Tailwind) via `create-next-app` into a scratch dir, then merged into the repo root (had to work around `create-next-app` refusing a non-empty target directory). Caught and fixed a self-inflicted bug immediately: the merge overwrote the project's governing `CLAUDE.md` with the scaffold's own generated `CLAUDE.md`/`AGENTS.md` stub ‚Äî recovered the original from `git show HEAD:CLAUDE.md` before continuing.
3. Installed Prisma, Tiptap, Zod, marked, DOMPurify, Vitest.
4. `prisma init` defaulted to Prisma 7, which requires driver adapters and rejects `datasource { url }` in schema ‚Äî hit `P1012` on first migration attempt. Downgraded to Prisma 6 (stable, standard config) rather than adopt the brand-new adapter pattern under time pressure.
5. Wrote the schema (`User`/`Document`/`Share`), seed script, `lib/prisma.ts`, `lib/auth.ts` (cookie-based mock session), `lib/access.ts` (single source of truth for owner/edit/view/none resolution), `lib/import.ts` (markdown/plaintext ‚Üí sanitized HTML), `lib/validation.ts` (Zod schemas).
6. Built all API routes (`/api/auth/*`, `/api/documents`, `/api/documents/[id]`, `/api/documents/[id]/share`, `/api/documents/import`) and all pages/components (login, documents list/board, share dialog, Tiptap-based document editor with toolbar and debounced autosave).
7. Installed `@tailwindcss/typography` for editor prose styling (Tailwind v4's CSS-based `@plugin` syntax, not the old JS config).
8. Ran `npm run build` clean, then started the dev server and drove the API directly with `curl` for every access-control case: owner CRUD, shared-editor edit, shared-viewer blocked from editing, owner-only delete/share enforcement, unauthenticated 401, cross-user 404 (a user with no share record gets the same "not found" as a nonexistent document ‚Äî no information leak about a doc's existence).
9. Tested file import: valid `.md` (headings/bold/lists converted), rejected `.pdf` (clear error message), and two XSS attempts (`<script>` tag in Markdown, `<img onerror>` in plaintext) ‚Äî confirmed DOMPurify/escaping neutralized both.
10. No `chromium-cli` available in this environment; wrote a throwaway Playwright script instead (per the `run` skill's documented fallback) to drive a real headless Chromium session end-to-end: login ‚Üí documents list ‚Üí open doc ‚Üí toolbar/content render ‚Üí type text ‚Üí autosave indicator ‚Üí share dialog. Checked `console --errors`-equivalent (page `console`/`pageerror` listeners) ‚Äî zero errors. Read the resulting screenshots to confirm visual layout.
11. Wrote unit tests: `lib/import.test.ts` (pure-function coverage of extension filtering, markdown conversion, plaintext escaping, XSS stripping) and `lib/access.test.ts` (mocked-Prisma coverage of the five access-resolution branches). Initially installed React Testing Library + jsdom for possible component tests; removed both once it was clear only logic tests were needed, to avoid shipping unused dependencies.
12. Started deployment: `vercel link` succeeded and auto-detected the project's existing GitHub remote (`charlesalo/docflow`). Began pursuing a Vercel Postgres/Neon integration to work around Vercel's serverless functions having a read-only filesystem (a real blocker for writing to a SQLite file there, not just an ephemeral-persistence risk) ‚Äî the candidate intervened here: the brief's "no paid dependency for reviewers" constraint was being over-satisfied by adding an unnecessary hosted-database integration when a deployment target with a real writable filesystem would meet the same constraint with fewer moving parts. Reverted to a plain-SQLite deployment plan; see `SUBMISSION.md` for the final target and result.
13. Deployed to Render via `render.yaml` blueprint (candidate connected the GitHub repo through their own dashboard ‚Äî account-gated step). First deploy failed: Render's `npm install` prunes `devDependencies`, so `@tailwindcss/postcss` and `typescript` ‚Äî both dev-only but required by `next build` ‚Äî were missing, breaking the Turbopack PostCSS step. Fixed by moving build-time packages into `dependencies` and excluding `vitest.config.ts`/`*.test.ts` from the tsconfig used by the build's type-check. Verified by simulating `NODE_ENV=production npm install && npm run build` in an isolated copy before re-pushing; second deploy succeeded.
14. Added username autocomplete to the share dialog: new `GET /api/users` (excludes self), filterable dropdown in `ShareDialog.tsx` with keyboard nav, already-shared users filtered out of suggestions.
15. Candidate reported a contrast bug (share-dialog names and document title "almost invisible"). Diagnosed as the browser's forced/auto dark-mode feature auto-inverting the page, since the app never declared a color-scheme. Fixed with `color-scheme: light` in CSS plus Next's `Viewport` export, rather than patching individual element colors. Verified the fix by actually launching Chromium with `--force-dark-mode --enable-features=WebContentsForceDark` and confirming the dialog still rendered in normal light contrast.
16. Made the per-collaborator permission pill in the share dialog an inline `<select>` (owner can flip a collaborator between edit/view without removing and re-adding them) ‚Äî candidate chose the dropdown pattern over a toggle via an explicit tradeoff discussion (dropdown matches the existing "add person" control and scales to a third permission tier later; a toggle would need to become an awkward multi-state control for that same case).
17. Built the one optional-stretch feature (candidate's pick from the brief's list, after a tradeoff discussion against real-time presence and Markdown export): document version history. New `DocumentVersion` table; checkpoints saved automatically after an edit, throttled to one per 3-minute window (`lib/versions.ts`, unit-tested), plus a manual "Save version now"; a `History` panel to preview and restore any checkpoint, gated through the same access-control model already built for sharing. Caught a real bug via curl-driven testing of the actual restore flow: the first implementation snapshotted state *before* an edit instead of after, so restoring a checkpoint returned stale/empty content rather than what was actually written ‚Äî fixed the ordering and re-verified with the same curl sequence plus a full browser pass before shipping.
